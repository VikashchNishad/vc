Why we need continuous integration ?

Before continuous integration the group of developers make changes to the repository (repository can be git repository or sub version repository) and when entire app is written to the git repository, built by Ant Maven etc. and after that the built application deployed on the test server for testing and if any bug in code the developer notify by feedback loop and if no any bug then application deployed on the production server for releases.

  • Developers have to wait till the complete software is developed for test results. Frist the entire source code of the application build then only deployed on the test server for testing. It takes lot of time. So developers have to wait long time in order to get the test result.
  • Since the entire source code of the application first built and then it is tested. So if test fails then locating and fixing bugs is very difficult. Developers have to check the entire source code of the software in order fix the bugs.
  • Since the developers wasting lots of time in locating and fixing bugs instead of building new application. So the software delivery process was slow.
  • Continuous feedback related to things like architecture issue, build failures, test status etc. was not present.

What is continuous integration ?

Continuous Integration is a development practice in which the developers are required to commit changes to the source code in a shared repository several times a day or more frequently. Every commit made in the repository is then built. After every commit to the source code an auto build is triggered then it is automatically deployed on the test server. If the test result shows that there is any bug in the source code at that time the developer is notified then the developer only have to check the last commit made to the source code in order to fix the bugs. Since the developers does not have to wasting lots of time in locating and fixing bugs now he can focus on the building new application. This increases the frequency of new software releases. It also provides continuous feedback related to things like architecture issue, build failures, test status etc.

Before continuous integration After continuous integration
The entire source code was built and then tested. Every commit made in the source code is built and tested.
Developers have waited for test result. Developers know the test result of every commit made in source code in the run.
No feedback. Feedback present.

Continuous integration tools

There are many tools that help in continuous integration. Jenkins, Buildbot, Travis and Bamboo are the most widely used continuous integration tools in market.

What is Jenkins ?

Jenkins is an open source automation tool written in Java with plugins built for Continuous Integration purpose. Jenkins is used to build and test your software projects continuously making it easier for developers to integrate changes to the project, and making it easier for users to obtain a fresh build. It also allows you to continuously deliver your software by integrating with a large number of testing and deployment technologies.
